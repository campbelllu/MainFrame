Funny Enough...

This project started as a homelab project to learn DevOps skills.

With a grasp on Docker, Terraform, AWS, CI/CD, we realized the next few steps involved deploying actual software. We could have grabbed something generic, made a hello world app, something,
but instead decided to start churning out something related to our current favorite hobby: Securities analysis.

Bouncing between multiple websites, trying to piece together the data we wanted, we thought, "Why not just build what we're looking for?"
So many months later, it's back to the DevOps side of things, getting ready to deploy version 0.0.1 of this thing.

Tech Stack

Python 3.10.12: Data cleaning done with Pandas library, coffee, sweat, tears, ecstatic joy each week as new learning hurdles were overcome
Django 4.2.1, with the expected gunicorn and NGINX setup
SQLite internal DB for now, chosen for simplicity and lack of user I/O, authentication, quick prototyping. Migration to PostgreSQL later if authentication is needed might be considered.
SEC's EDGAR API to source all data via requests library
Github Actions to full CI/CD at this time. Simple, familiar choice. May drop the CD part for another tool, learned recently that Ansible can actually be incorporated in this process which may prove helpful given we're deploying to...
AWS Lightsail for hosting. Cheap hosting option until migration due to scaling reasons becomes necessary.
Unsure at this point how new Lightsail instances and Terraform interact, how much maintenance is required for routing of new instances, setting them up to our NGINX proxy and with Route53, essentially, can we use Lightsail like we would EC2's, as cattle? Or are they a more static infrastructure piece and the headache wouldn't be worth the time? We'll see.
Vanilla HTML and CSS is the only 'secret sauce' running the front end at this point.

Flaws, Learning Opportunities, Vulnerabilities, Site-Self-Reflection

The admin page is completely disabled. A version of the Django secret key is in earlier repo versions, making Django Forms a security compromised Python tool. However, there is no sensitive data in the database and data tampering is nearly impossible. Even if the data were able to be accessed nefariously, we have backups, no sensitive data is present, and for much less of a hacking headache the data therein is already much more easily accessible via the EDGAR API link above. The juice is most definitely not worth the squeeze unless Django has a bug bounty out, in which case you're legally required to split the cash prize with us, the victim. :-p

If future versions of the website require login authentication, or the storing of any sensitive data like names, addresses, payment info, user passwords, anything at all, a complete overhaul of this app's security would be necessary. Django secret keys would be stored in a secrets vault, or in a .env file to be deployed and 'installed' as environment variables, properly securing the whole thing.

The UI consists solely of Django Templating: HTML + CSS + unholy amounts of time for our data analysts and backend engineers to figure out how frontend stuff works.

Future versions of the site, if UI revamp is necessary via increased traffic attention, will include Django Rest Framework, and most likely, a proper React frontend.
